# Design.md — Рекомендации по развитию Dungeon Card Crawler

Документ содержит рекомендации по масштабированию кор-механики, подготовке мета-геймплея и интеграции боссфайта в общий поток забега.

---

## 1. Архитектурное разделение: Run vs Meta

### 1.1. Два слоя игры

- **RunLayer (забег / уровень)** — отвечает за сетку, генерацию уровня, врагов, бой, движение, боссфайт, награды за конкретный забег. Живёт от старта уровня до победы/поражения.
- **MetaLayer (мета-прогресс)** — отвечает за валюты, апгрейды, разблокировки контента, статистику, выбор миссии/уровня. Живёт между сессиями, хранится в `localStorage`.
- **Связь через события**: RunLayer эмитит события (`RUN_ENDED`, `BOSS_KILLED`, `ITEM_PICKED`, `ENEMY_KILLED` и т.п.), MetaLayer подписывается и решает, как наградить игрока.

### 1.2. Структура состояний

- **RunState**: параметры игрока (HP/EN, инвентарь, оружие, позиция), сетка (rows), босс (статы, позиция, инвентарь), технические флаги (scrollY, анимации, текущий уровень).
- **MetaState**: ресурсы (валюта), апгрейды (дерево/список), разблокировки (уровни, враги, предметы), настройки. Хранится в `localStorage` (например, ключ `dcc_meta`).

## 2. Data-driven контент: регистры врагов и объектов

### 2.1. Регистры вместо жёстких if/else

- **Registry клеток**: `cellTypes[type] = { render(), onEnter(), blocksMovement, blocksLoS }` — добавление нового типа = запись в регистр, а не новая ветка в коде.
- **Registry врагов**: `ENEMY_DEFS[id] = { baseHp, visionRange, actionRange, label, color, aiProfile }` — новые типы врагов добавляются через данные.
- **Spawn-таблицы уровней**: развить текущие `chances` — несколько таблиц (ранние/средние/поздние ряды), модификаторы от меты.

### 2.2. Плюсы

- Добавление предмета/врага/эффекта = запись в конфиг + 1–2 функции, без переписывания ядра.
- Мета-геймплей может разблокировать типы через флаги в MetaState.

---

## 3. Управление состояниями экранов (state machine)

### 3.1. Явная машина состояний приложения

Состояния: `BOOT` → `META_HUB` → `RUN_PREPARE` → `RUN_PLAYING` → `RUN_SUMMARY` → `META_HUB`. Боссфайт — не отдельное состояние, а фаза внутри `RUN_PLAYING` (см. раздел 7).

### 3.2. Экраны как модули

Вынести в отдельные сущности: `LevelSelect / MetaHub`, `RunSummaryScreen`, `UpgradesScreen`.

---

## 4. Генерация уровней и отладка

### 4.1. Детерминированная генерация (Seed)

Для обеспечения воспроизводимости уровней и упрощения отладки, генерация игрового поля основана на начальном значении — **сиде** (seed).

- **Принцип работы**: В начале каждого забега инициализируется генератор псевдослучайных чисел (PRNG) с использованием сида. При одинаковом сиде генератор всегда будет выдавать одну и ту же последовательность чисел, что гарантирует идентичную генерацию уровня: расположение врагов, стен и бонусов будет одинаковым.

- **Как использовать**: Чтобы запустить игру с конкретным сидом, необходимо добавить параметр `?seed=<число>` в URL.
  - **Пример**: `http://127.0.0.1:5500/?seed=12345`

- **Случайная генерация**: Если параметр `seed` в URL не указан, игра автоматически сгенерирует случайный сид на основе текущего времени (`Date.now()`), обеспечивая уникальный забег каждый раз.

- **Отладка**: Текущий сид забега сохраняется в `runState.seed` и выводится в консоль разработчика при старте уровня. Если вы столкнулись с багом или интересной игровой ситуацией, просто скопируйте этот сид и используйте его в URL для повторного воспроизведения.

- **Сброс**: При выходе из забега в главное меню параметр `seed` автоматически удаляется из URL, чтобы следующий запуск по умолчанию был случайным.

### 4.2. Контракты вместо простого выбора уровня

Уровень описывается не только `rows` и `chances`, но и мутаторами (сложность, награды) и наградой (множитель валюты, редкие ресурсы).

## 5. Оптимизация производительности

### 5.1. Оконный рендер (render window)

Рендерить только видимые ряды: `fromY = playerY - 1`, `toY = playerY + VISIBLE_ROWS + 2`. Не проходить по всем `gameState.rows` каждый кадр.

### 5.2. Локальное обновление врагов

Обновлять состояния врагов (ALERT и т.п.) только для ряда игрока и соседних рядов, не для всего уровня.

### 5.3. UI по событиям

Вызывать `renderInventory()` и подобные функции по событиям (`INVENTORY_CHANGED`, `WEAPON_CHANGED`), а не каждый кадр в `update()`.

---

## 6. Идеи мета-геймплея

### 6.1. Мета-валюта и награды

- Валюта за забег: базовая награда за босса/пройденные ряды, бонусы за «чистоту» (мало урона, экономия EN).
- Использование: покупка апгрейдов, разблокировка контента.

### 6.2. Базовое дерево апгрейдов

Примеры: «Усиленный корпус» (+HP), «Реактивный источник» (+EN), «Усиленный арбалет» (+урон), «Дополнительный слот атаки/защиты», «Тактический запас» (стартовые бустеры).

### 6.3. Хаб вместо меню уровней

Экран с деревом апгрейдов, выбором контракта/миссии, статистикой. Текущий «выбор уровня» становится частью хаба.

---

## 7. Боссфайт в общем потоке забега

Боссфайт интегрируется в забег без отдельного стейта. Оба режима — одна модель: сетка, ходы, инвентарь, бой. Отличаются только направление хода и очередность действий.

### 7.1. Один appState и фаза уровня

Вместо `appState: 'bossfight'` использовать:

```
levelPhase: 'dungeon' | 'boss_arena'
```

`appState` остаётся `'playing'` всё время. Босс — часть уровня (последние 2 ряда).

### 7.2. Общая логика «куда можно ходить»

Одна функция `getValidTargetCells()` возвращает клетки в зависимости от фазы:

- **dungeon**: клетки в ряду `player.y + 1` (следующий ряд вверх);
- **boss_arena**: клетки в ряду `player.y` (текущий ряд, горизонталь), кроме текущей позиции.

Один обработчик клика: клик по валидной клетке → `processMove(targetX, targetY)`.

### 7.3. Очередность ходов

В dungeon враги реагируют при уходе с ряда — это «реакция», не отдельный стейт.

В boss_arena после хода игрока нужен ход босса. Достаточно флага:

```
turnOwner: 'player' | 'boss'
```

- `turnOwner === 'player'` — обрабатываем клики;
- `turnOwner === 'boss'` — блокируем ввод, запускаем AI босса, по завершении ставим `turnOwner = 'player'`.

Это подсостояние внутри `playing`, а не новый стейт приложения.

### 7.4. Единый поток обработки

Свести `handleInput` (dungeon) и `processPlayerBossTurn` (boss) к одному потоку:

1. Клик → `getValidTargetCells()` → клетка в списке?
2. Если да → `processMove(gx, gy)`:
   - расход энергии (общая формула);
   - `checkAttackOnLeave()` (в dungeon — враги, в boss_arena — не используется);
   - перемещение + `processInteraction(cell)` (общая функция);
   - если `levelPhase === 'boss_arena'` → `turnOwner = 'boss'`, `scheduleBossTurn()`.

`processInteraction` уже общая: HEAL, ENERGY, ATTACK_BOOST и т.д. работают одинаково. Отдельная логика только для BOSS_DAMAGE и ближнего боя с боссом — её держать внутри `processInteraction` по типу клетки и наличию босса рядом.

### 7.5. Рендер и UI

Рендер уже общий: сетка, игрок, босс, инвентарь. Разница только в размере тайлов для арены — задаётся через `levelPhase` при расчёте координат.

### 7.6. Итог по боссфайту

Боссфайт — часть забега, а не отдельный режим. Ключевые изменения:

- `levelPhase` и `turnOwner` вместо `appState === 'bossfight'`;
- `getValidTargetCells(levelPhase, playerPos)` для валидных клеток;
- единый `processMove` с ветвлением по фазе;
- `processInteraction` общая, с особыми случаями для босса.

---

## 8. Пошаговый план внедрения

1. **Рефактор ядра**: вынести код из `index.html` в модули, ввести RunState/MetaState, стейт-машину экранов.
2. **Registry и оконный рендер**: registry клеток/врагов, рендер только видимого окна.
3. **MetaLayer**: MetaState с валютой и апгрейдами, Run Summary экран, хаб вместо меню уровней.
4. **Контракты и контент**: seed-генерация, мутаторы, расширение пула врагов/предметов через регистры.
